# Individual Reflection — Derek Zhang

When I started Prova, I thought the hard part would be getting three AI agents to produce reliable compliance assessments. It turned out the hard part was everything around them — the orchestration, the scoring integrity, the CI pipeline, and figuring out how to review 400 lines of AI-generated code without losing focus. Building a production app with Claude Code taught me that the AI is the easy part. The engineering discipline around it is what makes or breaks the project.

The most impactful lesson was learning to split work by concern. Early on, I gave Claude Code an entire issue — API routes, database queries, frontend pages, modals — and got back a massive diff that was technically correct but impossible to review thoroughly. With ADHD, I know my attention degrades on long review sessions. So I started splitting every full-stack issue into a backend run and a frontend run. Each diff was 200 lines instead of 500. Each review session had a clear scope. This single workflow change — which I later formalized in the implement-feature v2 skill — was the most valuable thing I did all project.

The plan-before-execute pattern saved me from at least two architectural mistakes. On S2-06, the planning step caught a data shape mismatch between what the ComplianceResults component expected and what the submission API actually returned. Without that planning pause, Claude Code would have built the wrong abstraction, and I would have spent an hour debugging why the submission detail page was rendering empty pillar cards. Instead, we caught it before a single line of code was written and chose the right approach — reusing sub-components directly instead of faking a data shape we did not have.

The CI pipeline was another turning point. Before S3-01, errors only surfaced when I ran npm run build locally. After adding GitHub Actions, the pipeline immediately caught lint errors that build had ignored — a setState-in-useEffect violation and several unused variables. Both the CI/CD PR and the Help Page PR failed on first push. I had to create a separate fix PR for main, merge it, then rebase four feature branches. That experience is why the v2 skill now mandates running lint, typecheck, and build before every commit. Since adding that rule, I have had zero CI failures from lint issues.

Looking back, the project taught me that AI-assisted development is not about typing less — it is about thinking more carefully about what you ask for and reviewing what you get back. The workflow I built — design in Claude chat, craft precise prompts, execute in Claude Code, review the output, iterate — is a genuine engineering process. The AI handles the volume. I handle the judgment. That division of labor worked, and it is a pattern I will carry forward into every project after this one.
