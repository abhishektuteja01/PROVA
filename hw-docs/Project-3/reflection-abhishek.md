# Individual Reflection — Abhishek Tuteja

I came into this project with a specific frustration. I had been reading SR 11-7 documentation and kept thinking about how mechanical the compliance review process is — junior validators going through the same 20 elements with the same checklist, document after document. It felt like exactly the kind of problem where AI could do the boring part and let humans focus on judgment. That frustration became Prova.

The first real decision was architectural, and it shaped everything that followed. I tried a single-agent approach first — one prompt, one model, assess everything. The results were mediocre. The agent would start strong on the first few elements and then get lazy toward the end, skipping details or producing vague assessments. So I split it into three specialist agents, one per SR 11-7 pillar, running in parallel. Each agent only knows about its own elements. It cannot cut corners on element seven because it is tired from elements one through six. That decomposition — giving each agent a narrow, well-defined scope — was the single best design decision in the project.

The thing that surprised me most was how often the agents got their own scores wrong. An agent would find five Critical gaps and three Major gaps, then report a score of 25. The math says 0. This happened on nearly every run. I built `verifyPillarScore` to recalculate scores from the gap list and override whatever the agent claimed. We log the discrepancy but never use the agent's number. That taught me something I keep coming back to: language models are good at qualitative assessment — finding gaps, classifying severity, writing summaries — but you should never let them do arithmetic. Do the math yourself.

The CLAUDE.md system became unexpectedly important. I started it as a simple rules file, but by Sprint 3 it had grown into a modular system with 10 sub-files imported via `@imports`. Each folder has its own CLAUDE.md with domain-specific rules — the agents folder knows about bias mitigation and prompt injection, the scoring folder knows the formula, the API folder knows the route contracts. When Claude Code works on a file, it automatically gets the right context without me having to explain anything. The rules compound — new sessions inherit months of accumulated decisions. That compounding effect is the real value of CLAUDE.md, not the individual rules themselves.

The hardest sprint was Sprint 3. I spent four days fighting `pdf-parse` — it needed DOM polyfills that broke on Vercel's serverless runtime. I went through six PRs, two reverts, and eventually ripped it out entirely and switched to `unpdf`. The lesson was painful but clear: evaluate serverless compatibility before adopting a library, not after you have built your entire file parsing pipeline around it.

If I had to summarize what I learned, it is this: building with AI is not a shortcut. It is a different kind of engineering. You still need architecture. You still need testing. You still need to review every line. The AI generates code faster than I can type it, but the bottleneck was never typing — it was knowing what to build and verifying that what got built was correct. Prova works because we treated Claude Code as a fast junior engineer who needs clear instructions, tight scope, and constant review. Not as magic.
